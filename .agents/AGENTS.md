# Project-Scoped Rules for KnittingProject

## Architecture and State Management
- **Stateless Core**: `knitting_core.py` MUST remain a stateless, purely functional module. Do not introduce classes that hold state. Functions in this module should take their necessary parameters as arguments and return the computed results.
- **Centralized State**: All application state must reside in `app_state.py` within the `AppState` class (using its `self._data` dictionary). Do not spread state variables or instantiate stateful objects across different files (`app.py`, `gui.py`, or `knitting_core.py`).
- **No Side Effects in Core**: `knitting_core.py` should not perform side effects like resolving file paths or reading from the filesystem. These operations belong to `app_state.py` or the application entry points (`app.py`).
- **Data-Driven and DRY Design**: Prefer declarative, data-driven approaches over boilerplate. Use dynamic schemas, configuration objects, and centralized defaults instead of writing repetitive getter/setter wrappers or hardcoded properties. When loading configuration schemas or metadata, bind and map properties dynamically (e.g., looping over keys) rather than writing lines of sequential manual assignments (e.g., repeating `setattr` or `super().__setattr__` calls). Proactively refactor any existing boilerplate of this nature to use dynamic loops.
- **Systemic Refactoring (No Half-Measures)**: When instructed to remove a block of code, feature, or component, always trace and eliminate all of its cascading usages, orphaned variables, leftover imports, and dependencies across the entire project. Do not stop until the job is fully finished and the codebase is clean.
- **Pure Spline Geometry Pipeline**: The application utilizes spline-based mesh reconstruction (`build_spline_mesh`) as its canonical geometry pipeline. Avoid re-introducing or restoring the deprecated parameter-based vertex evaluation routines (`compute_knitting_vertices`, `rebuild_param_mesh`) to maintain low structural density and keep the codebase clean of dead code.
- **No Backward Compatibility Branching**: Do not implement branching, validation wrappers, or fallback layers in code to support deprecated data versions or legacy schemas. Maintain linear, non-branching execution paths. If data formats or structures change, manually update or migrate the input data files (e.g. `params.json`) rather than writing adaptive code.
